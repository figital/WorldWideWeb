Performance of FIND server 29 May 91

Conditions: Loops over XFIND COPY TAPE CMS with SHOW DATE. Normally loaded machines (around 14:00) running VMS and accessing CRNVMC with HTTP/TCP/IP. Loops/min Loops/sec One process on one machine at a time: VXCRNA 27 .45 Hz ONLINE 27 .45 Hz One process on each machine, all at the same time: VXCRNA 25 .4 Hz ONLINE 20 VSOC28 18 VSOC06 16 Total 79 1.3 Hz Thee spawned processes each machine, all the same time (12 in all): VXCRNA 49 .4 Hz ONLINE 21 VSOC28 20 VSOC06 19 Total 109 1.8 Hz The last test apparently brought the VM TCP interface close to running out of buffers and losing telnet connections. This is with a version of the daemon which incorrectly gives precedence to incoming new connections over messages on existing connections, and so the number of connections at any one time may have been larger than necessary. The test demonstrates that the server (if not the IBM) behaves well under overload. The throughput figure is sufficient for our estimates of peak load on a running system.

From Feb 29 meeting:

See browser outline module dependency diagram we discussed.
  • Does the Mac environment allow malloc()? If not, we have to make a malloc().
  • Does a Mac socket() return something which can be used like a file? Maybe anyway (for TWG) we have to make a HTStream macro a la NXStream. The main operations on it would be block reads and writes.
  • The object-oriented parts of Think-C can't be used within hypercard. This suggests use c++ stand-alone on the Mac or we use just C all over, in OO style. An object which accepts the begin/end style, begin/end anchor, and add text functions (methods) seems to recur. Tim would next time rewrite the Anchor handling with subclasses for parent and child anchors.
  • We should get LaTeX manuals, to see about conversion.
  • We must look at Interviews and Andrew.
  • Should we make a portable text object? If so, should we make it as simple as possible, or make it a balanced/structured btree including modification items and structured text? Where do we draw the line? We must take into account writing time, maintainability, execution time (including program+data size which affects paging and is limited on the Mac).
